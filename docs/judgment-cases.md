# 判斷案例庫（judgment-cases）

目的：累積本專案「判斷錯了被糾正、或差點判斷錯」的實例。規範檔（AGENTS.md）教原則，
這裡放帶情境的實例——AI 代理在遇到相似情境時，套用案例比套用原則命中率更高。

**維護規則**：每次發生「模型判斷錯誤被使用者糾正」或「合理假設被驗證推翻」，就加一條。
一條案例 = 情境 + 當時的判斷 + 裁決結果 + 可轉移的教訓。有數字就附數字，禁止事後美化。

---

## 案例 1：合理的假設也會被回測否決（Adaptive VOL_TARGET，2026-07-02）

- **情境**：假設曝險模型長期低估水位（premise：實現波動率中位數約 27%），提出用滾動 1 年波動率分位數自適應調整 VOL_TARGET。
- **當時的判斷**：邏輯上說得通，實作也完成了。
- **裁決**：3y/5y 回測——台股 Sharpe 只有 1.82→1.87 / 1.30→1.33 的微幅改善，美股反而劣化 1.43→1.37 / 0.65→0.58；且前提本身就是錯的（滾動中位數實際落在 15–20%，不是 27%）。功能保留在 `VOL_TARGET_ADAPTIVE=False` 開關後，不啟用。
- **教訓**：「邏輯合理 + 實作完成」不構成採納理由，回測數字才是裁決。**否決的實驗要把結論記錄在決策點**（config 註解、commit message），防止未來的代理或人重提同一假設。重新驗證用 `tests/backtest_exposure_3y.py --compare`。

## 案例 2：別的市場的結論不能直接外推（台股頂部排序，2026-07-02）

- **情境**：美股回測已證明高周轉排序損害 Sharpe（納入短期訊號 1.41→1.14），據此推論台股「按當日漲幅排序取頂部」也該改成按總分排序。
- **當時的判斷**：既然美股如此，台股照改即可——看起來不用再驗。
- **裁決**：`tests/backtest_tw_ranking.py` 用生產版評分逐週重演 45 個評分日：現行的漲幅排序在 5/10/20 日全部視窗都勝過總分排序（例：20 日 12.8% vs 11.0%）。現行邏輯保留。
- **教訓**：同一假設在不同市場（策略、資料頻率、參與者結構都不同）必須各自回測。**「在 A 成立」只是「在 B 值得一驗」的理由，不是「在 B 直接改」的理由。**

## 案例 3：回測異常先驗資料、再驗邏輯（cache 過期）

- **情境**：回測結果與預期不符時，直覺反應是調參數或改邏輯。
- **正確順序**：`src/advisor/cache/` 的 pkl 帶日期後綴（如 `hist_2y_20260703.pkl`），先確認腳本讀到的是當期 cache、資料區間涵蓋預期範圍，再去動邏輯。資料錯，後面全部白做。
- **教訓**：診斷順序固定為 **資料 → 讀取 → 邏輯 → 參數**。連調兩次參數無效，就該回頭驗資料（見 AGENTS.md「方向錯了的訊號」）。

## 新增案例模板

```markdown
## 案例 N：一句話標題（主題，YYYY-MM-DD）

- **情境**：當時面對什麼問題、有什麼背景
- **當時的判斷**：模型（或人）原本怎麼判斷、為什麼那樣想
- **裁決**：實際驗證的結果，附數字或輸出
- **教訓**：可轉移到未來相似情境的一句話判準
```

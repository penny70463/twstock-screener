# AGENTS.md — 跨工具代理行為準則（twstock-screener）

本檔是所有 AI 編碼工具（Claude Code、Gemini CLI、Cursor 等）的共同行為準則，
與專案技術說明（CLAUDE.md）互補：CLAUDE.md 講「這個專案長什麼樣」，本檔講「怎麼判斷與驗證」。
內容改寫自個人工作制度（judgment-rubrics），範例已換成本專案情境。

## 語言偏好

- 對話回覆一律使用繁體中文
- Git commit message 一律使用英文

## Definition of Done（改動代碼後，全部通過才算「完成」）

**機械化入口：宣告完成前必跑 `scripts/verify_done.sh`，任何 FAIL 都不得宣告完成。**
腳本會自動偵測改動範圍並執行對應檢查（語法、離線測試、results 防護、回測證據、dry-run）；
改了 `src/advisor/config.py` 時需以 `BACKTEST_EVIDENCE=<回測輸出檔>` 提供證據。
腳本涵蓋不到的判斷（子項覆蓋、回報誠實度）仍依下列條文：

1. **診斷乾淨**：改動的每個檔案 IDE diagnostics / linter 無 error（無 IDE 診斷時以 `python -m py_compile <檔案>` 為底線）
2. **策略參數改動必回測**：改了 `src/advisor/config.py` 的任何參數，必須跑對應的 `tests/backtest_*.py` 並在回報中附上前後對比數字——「參數看起來合理」不算驗證
3. **實跑過受影響流程**：
   - 篩選邏輯改動 → `python run_pipeline.py`（或針對性測試）跑通
   - 警報邏輯改動 → `python etf_alert.py --dry-run`（絕不在驗證時真發 LINE）
4. **需求的每個子項**都有對應改動，或明確說明為何不做
5. **不手改 `data/results/`**：結果 JSON 由管線產生並自動 commit/push，手動修改會被下次執行覆蓋且污染歷史

## 何時該停下來問使用者

滿足任一條就停下來問，其餘情況自行決定並在回報中說明理由：

- 動作不可逆或影響對外狀態：真的發送 LINE 推播、git push、刪除 cache 或歷史結果檔
- 改動策略參數（評分權重、門檻、曝險目標）但沒有回測依據——回測是驗證手段，不是決策授權；影響實際操作的參數方向要使用者拍板
- 兩種做法都合理但影響差異大（例：修資料異常用清洗補丁 vs 改上游抓取邏輯）
- 需求有歧義且猜錯成本高（例：「排名怪怪的」——是評分權重、產業上限、還是快取過期？先查證，查完仍有多種解讀才問）

反例（不用問，直接做並在回報提一句）：改變數名要不要順手改測試引用、補漏掉的 docstring——把這種問題丟回給使用者是浪費時間。

## 方向錯了的訊號（該換路，而非重試）

出現任一訊號，停止當前方法，退回上一個決策點換方法：

- 同一個錯誤在兩次不同修法後仍出現 → 你對錯因的假設是錯的，重新診斷而不是第三次修
- 修好 A 冒出 B、修好 B 冒出 C → 改動層級錯了，往上找共同根因
- 發現自己在對抗既有機制（例：繞過 TWSE/TPEX 內建的指數退避直接重試、在 screener 外面再包一層過濾去修 screener 內的 bug）→ 停，用正規切入點
- 為了一個小需求要改 10 個以上不相關檔案 → 幾乎肯定有更小的切入點，先找現有機制
- 同一件事、同一方法，最多重試兩輪。第三次原樣重試是被禁止的——必須換方法、換工具、或如實回報卡點

正例：回測結果與預期不符，第一次調參數沒效、第二次再調也沒效 → 停止調參，改為檢查回測腳本讀的資料區間與 cache 是否過期（先驗資料，再驗邏輯）。

## 驗證紀律（做的人不驗收自己的工作）

驗證手段要和風險成比例：

- 檔案類產出 → read-back：實際打開檔案抽查關鍵段落與結尾，確認沒被截斷
- 代碼改動 → 實跑或測試優先，diagnostics 是底線不是全部
- 涉及金額計算、策略參數、對外推播 → 用回測數字或 --dry-run 輸出作證據，不接受「邏輯上應該對」
- 批次改動（多檔同構修改）→ 寫一行腳本機械驗證完整性，不要人肉逐檔看
- 資料類結論（某股票分數、某日曝險值）→ 附上來源（哪個 JSON、哪次執行的輸出），區分「確認的事實」與「推測」

## 誠實條款

- 品味與模糊判斷（策略取捨的長期代價、UI 呈現）：給 2–3 個帶取捨說明的選項讓使用者選，明說「這是判斷題，我給選項不替你決定」
- 能力不及的問題：如實報告卡點與已試路徑，建議外部資源，不要重試到 context 耗盡
- 不確定的事就查；查不到就標註「未驗證」；禁止編造（尤其是財務數據與回測結果）
- 測試失敗就說失敗並附輸出；跳過的步驟明說跳過

## 族群（題材）分類

凡涉及「個股分族群／題材分類／族群突破分析」，一律遵循 `docs/cluster-classification-guide.md`：
題材 ≠ 官方產業別、分到供應鏈環節顆粒度、2 檔同質即可成群、不同環節不可硬併。
程式化分類直接重用 `src/prompts.py` 的提示詞與 `src/classifier.py`，不要自行重寫規則；
改分類原則時，本指引與 `src/prompts.py` 必須同步修改。

## 判斷案例庫

`docs/judgment-cases.md` 收錄本專案真實發生過的判斷錯誤與裁決結果（附回測數字）。
遇到「要不要改參數／要不要外推結論／回測結果怪怪的」這類情境時，先查案例庫有沒有相似前例。
你的判斷被使用者糾正、或合理假設被驗證推翻時，主動在案例庫加一條（照檔尾模板）。

## 專案特定陷阱（判斷時的已知地雷）

- TWSE/TPEX API 有速率限制，重試邏輯已內建指數退避——不要繞過
- LLM 題材分類批次上限 30 股/請求（CI 270s 超時限制）
- `src/advisor/cache/` 下的 pkl 有日期後綴，回測前先確認 cache 是否為當期資料
- 前端 localStorage 存用戶持股，任何前端改動不得引入上傳行為
